# Phase 6 — Daily Content Refresh + Per-User DailyContent Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Promote the Phase 3 fixture-backed daily content into a real per-user, per-day DB-backed resource. Add a `DailyContent` Prisma model, two server-side primitives (`setDailyContent`, `getDailyContent`), a hand-edit modal so the user can paste/edit raw JSON, and a stub "Refresh today's content" button that Phase 8 will wire to an LLM. The contract that Phase 3 set up — `getDailyContent(userId, iso): Promise<DailyContent>` — does not change for any consumer; only the body of `src/server/queries/daily-content.ts` flips from "always return fixture" to "return DB row, fall back to fixture."

**Architecture:**
- One new Prisma table: `DailyContent(id, userId, iso, contentJson, source, createdAt, updatedAt)`, unique on `(userId, iso)`. `source` is `'manual' | 'llm' | 'fixture'`. Migration named `add_daily_content`. `contentJson` is stored as a string column on SQLite (`String`) and parsed at the boundary, so the same Prisma schema works on both SQLite (local) and Postgres (Railway, Phase 13). The runtime shape is enforced by `DailyContentSchema` from Phase 3 — Prisma does not validate it.
- Two server functions live next to the existing query file:
  - `getDailyContent(userId, iso)` — DB lookup; on miss, return `fixtureFor(iso)` and tag the result so the dashboard can show an empty-state hint. Public consumers get a `DailyContent` (the same shape as Phase 3); a sibling helper `getDailyContentWithMeta` returns `{ content, source }` for the dashboard hero.
  - `setDailyContent(userId, iso, json, source)` — `'use server'` action. Validates `json` against `DailyContentSchema` first; on failure, throws a `DailyContentValidationError` carrying a human-readable list of paths + messages (which property/index failed) — never raw Zod output. On success, upserts the row and `revalidatePath("/")`.
- The hand-edit UI is one Client Component (`HandEditModal`) wired to a Server Action (`saveDailyContentAction`). The modal opens from a small "edit content" link in the topbar (Phase 1 already reserves a slot next to the theme toggle). The "Refresh today's content" button lives inside the same modal as a STUB — disabled, with a tooltip pointing at Phase 8.
- Empty-state hint: when `getDailyContentWithMeta` returns `source === 'fixture'`, the hero subtitle gains a small "Showing default content — paste or generate yours" line. Once a row exists for `(userId, todayISO())`, the hint disappears.
- All mutations go through server actions; `getCurrentUserId()` (Phase 7 preview already established in Phase 2) resolves the user. Phase 6 does not introduce auth changes.

**Tech additions this phase:** none. Phase 3 already brought in Zod and the schema. We use Next.js `revalidatePath` (already in the stack), Prisma migrations (already in the stack), and a tiny client modal built with `useState` + the existing `<dialog>` styling pattern from `BibleModal`.

**Why a one-file query change works.** Phase 3 deliberately wrote every consumer (`MindfulnessPanel`, `BusinessPanel`, `PersonalPanel`, `Hero`) against `getDailyContent(userId, iso)`. Phase 6 swaps the body of that function. The only file that grows the contract is the same query file, where we add `getDailyContentWithMeta` for the empty-state hint — a strictly additive change.

```prisma
//   dailyContents DailyContent[]

model DailyContent {
  id          String   @id @default(cuid())
  userId      String
  iso         String   // ISO date "YYYY-MM-DD"
  contentJson String   // serialized DailyContent JSON; validated at the boundary
  source      String   // 'manual' | 'llm' | 'fixture'
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, iso])
  @@index([userId, iso])
}
```

Notes for the executing agent:
- `contentJson` is `String` (not `Json`) so the same schema compiles for SQLite and Postgres without provider-specific gates. Phase 13 may switch to `Json` on Postgres later; for now we parse with `JSON.parse` at the boundary.
- `source` is a free string, not a Prisma enum. SQLite has no native enum and we don't want a provider-specific schema for v1. Validation is enforced at the Zod boundary in `setDailyContent`.
- `onDelete: Cascade` matches the pattern used by `Goal`, `Task`, etc. in Phase 2.

- [ ] **Step 2: Implement `src/lib/format-zod-error.ts`**

```ts
import type { ZodError, ZodIssue } from "zod";

const MAX_ISSUES = 10;

function pathToString(path: ZodIssue["path"]): string {
  if (path.length === 0) return "(root)";
  let out = "";
  for (const seg of path) {
    if (typeof seg === "number") out += `[${seg}]`;
    else if (out === "") out = String(seg);
    else out += `.${String(seg)}`;
  }
  return out;
}

export function formatZodError(err: ZodError): string {
  const issues = err.issues.slice(0, MAX_ISSUES);
  const lines = issues.map((i) => `${pathToString(i.path)}: ${i.message}`);
  if (err.issues.length > MAX_ISSUES) {
    lines.push(`…and ${err.issues.length - MAX_ISSUES} more`);
  }
  return lines.join("\n");
}
```

```ts
import "server-only";
import { db } from "@/server/db";
import { DailyContentSchema, type DailyContent } from "@/types/daily-content";
import { fixtureFor } from "@/lib/daily-content-fixture";

export type DailyContentSource = "manual" | "llm" | "fixture";
export type DailyContentWithMeta = {
  content: DailyContent;
  source: DailyContentSource;
};

/**
 * Phase 6: read the per-user, per-day row from the DailyContent table.
 * Falls back to the fixture only when no row exists for that (userId, iso) —
 * this preserves Phase 3 behavior for users who haven't generated content yet
 * and gives new users a sensible starter on day one.
 *
 * NOTE: the row's `contentJson` is re-validated against DailyContentSchema on
 * read. If a corrupt row sneaks in (e.g. a manual SQL edit), we log and fall
 * back to the fixture rather than crashing the dashboard.
 */
export async function getDailyContent(userId: string, iso: string): Promise<DailyContent> {
  const meta = await getDailyContentWithMeta(userId, iso);
  return meta.content;
}

export async function getDailyContentWithMeta(
  userId: string,
  iso: string,
): Promise<DailyContentWithMeta> {
  const row = await db.dailyContent.findUnique({
    where: { userId_iso: { userId, iso } },
  });

  if (!row) {
    return { content: fixtureFor(iso), source: "fixture" };
  }

  let parsed: unknown;
  try {
    parsed = JSON.parse(row.contentJson);
  } catch (e) {
    console.error("[daily-content] row has invalid JSON, falling back to fixture", {
      userId, iso, error: (e as Error).message,
    });
    return { content: fixtureFor(iso), source: "fixture" };
  }

  const result = DailyContentSchema.safeParse(parsed);
  if (!result.success) {
    console.error("[daily-content] row failed schema validation, falling back to fixture", {
      userId, iso, issues: result.error.issues.length,
    });
    return { content: fixtureFor(iso), source: "fixture" };
  }

  const source = isValidSource(row.source) ? row.source : "manual";
  return { content: result.data, source };
}

function isValidSource(s: string): s is DailyContentSource {
  return s === "manual" || s === "llm" || s === "fixture";
}
```

- [ ] **Step 2: Create `src/server/actions/daily-content.ts`**

```ts
"use server";

import { revalidatePath } from "next/cache";
import { db } from "@/server/db";
import { getCurrentUserId } from "@/server/auth-context";
import { DailyContentSchema } from "@/types/daily-content";
import { formatZodError } from "@/lib/format-zod-error";
import type { DailyContent } from "@/types/daily-content";

export class DailyContentValidationError extends Error {
  readonly kind = "DailyContentValidationError" as const;
  constructor(message: string) {
    super(message);
    this.name = "DailyContentValidationError";
  }
}

/**
 * Validates the supplied JSON against DailyContentSchema and upserts the row.
 * Throws DailyContentValidationError with a human-readable, multi-line message
 * on schema mismatch (never raw Zod output).
 *
 * `source` defaults to 'manual'. Phase 8 calls this with 'llm' from the LLM
 * adapter; Phase 10's scheduler also calls it with 'llm'.
 */
export async function setDailyContent(
  userId: string,
  iso: string,
  json: unknown,
  source: "manual" | "llm" = "manual",
): Promise<DailyContent> {
  const result = DailyContentSchema.safeParse(json);
  if (!result.success) {
    throw new DailyContentValidationError(formatZodError(result.error));
  }
  const data = result.data;

  // Defensive: the embedded date field must match the iso row key. Otherwise
  // a stale "yesterday" payload could overwrite "today" silently.
  if (data.date !== iso) {
    throw new DailyContentValidationError(
      `date: payload date (${data.date}) does not match the day being saved (${iso})`,
    );
  }

  await db.dailyContent.upsert({
    where: { userId_iso: { userId, iso } },
    create: {
      userId,
      iso,
      contentJson: JSON.stringify(data),
      source,
    },
    update: {
      contentJson: JSON.stringify(data),
      source,
    },
  });

  return data;
}

/**
 * Form-action wrapper consumed by HandEditModal. The textarea posts a single
 * `json` field as a string. We parse, then delegate to setDailyContent.
 *
 * Returns { ok: true } on success, { ok: false, error: string } on failure —
 * the modal displays the error inline rather than crashing the page.
 */
export async function saveDailyContentAction(
  iso: string,
  rawJson: string,
): Promise<{ ok: true } | { ok: false; error: string }> {
  let parsed: unknown;
  try {
    parsed = JSON.parse(rawJson);
  } catch (e) {
    return {
      ok: false,
      error: `(root): not valid JSON — ${(e as Error).message}`,
    };
  }

  const userId = await getCurrentUserId();

  try {
    await setDailyContent(userId, iso, parsed, "manual");
  } catch (e) {
    if (e instanceof DailyContentValidationError) {
      return { ok: false, error: e.message };
    }
    throw e;
  }

  revalidatePath("/");
  return { ok: true };
}
```

The modal opens with the current day's JSON pretty-printed in a textarea. The user edits, clicks Save, and:
- on success: the modal closes with a small "Saved" line, and the dashboard re-renders via `revalidatePath("/")`.
- on validation failure: the textarea stays open, and a red panel underneath lists the failing paths (one per line).

The "Refresh today's content" button is rendered next to "Save" but is disabled with a `title=""` tooltip pointing at Phase 8.

- [ ] **Step 1: Create `src/components/content/HandEditModal.tsx`**

```tsx
"use client";

import { useState, useTransition } from "react";
import type { DailyContent } from "@/types/daily-content";
import { saveDailyContentAction } from "@/server/actions/daily-content";

type Props = {
  iso: string;
  initialContent: DailyContent;
  open: boolean;
  onClose: () => void;
};

export function HandEditModal({ iso, initialContent, open, onClose }: Props) {
  const [text, setText] = useState(() => JSON.stringify(initialContent, null, 2));
  const [error, setError] = useState<string | null>(null);
  const [savedAt, setSavedAt] = useState<string | null>(null);
  const [pending, startTransition] = useTransition();

  if (!open) return null;

  const onSave = () => {
    setError(null);
    setSavedAt(null);
    startTransition(async () => {
      const r = await saveDailyContentAction(iso, text);
      if (r.ok) {
        const t = new Date().toLocaleTimeString();
        setSavedAt(t);
        // Brief delay so the user sees confirmation before the modal closes.
        setTimeout(() => onClose(), 600);
      } else {
        setError(r.error);
```

## Notes for the agent executing this plan

1. **The single-file contract change is the whole point.** Phase 3 routed every consumer through `getDailyContent(userId, iso)`. Phase 6 swaps the body of that function to read from the DB with a fixture fallback. **Do not** change any Phase 3 consumer files (`MindfulnessPanel`, `BusinessPanel`, `PersonalPanel`, hero/scripture/etc.) except the additive `Hero.tsx` change which simply opts into the new `getDailyContentWithMeta` to render the empty-state hint. If you find yourself editing more than the files in the table at the top of this plan, stop and reassess — you have probably misread the contract.

2. **`DailyContent` is per-user, not shared.** Even in single-user / `AUTH_MODE=none` local mode, the row is keyed by `local-default`'s user id. This matters for Phase 13 (Railway, multi-user) where two users on the same deployment must see independent content. The integration test "isolates rows per user" pins this contract.

3. **The fixture file does not go away.** `src/lib/daily-content-fixture.ts` stays in the tree and continues to be the source for users who have no DB row yet. It is also the seed for the modal's textarea on the very first edit (because the modal's `initialContent` is whatever `getDailyContent` returns — the fixture, if no row exists). Treat the fixture as a "blank slate template" the user customises by pasting/editing.

4. **`source` semantics drive Phase 8 and Phase 10.**
   - `'fixture'` is **never written** by application code (the schema allows it for forward compat / debugging seeds, but no code path emits it). It only appears as a transient label from `getDailyContentWithMeta` when there is no row.
   - `'manual'` is what `saveDailyContentAction` writes — the user pasted JSON.
   - `'llm'` is what the Phase 8 LLM adapter and the Phase 10 scheduler write. Phase 8 will call `setDailyContent(userId, iso, generatedJson, 'llm')`.
   - The dashboard uses `source` only for the empty-state hint; the rest is bookkeeping for the refresh log Phase 10 builds on top.

5. **Phase 8 (LLM) hand-off.** Phase 8 imports `setDailyContent` directly. The "Refresh today's content" stub button in this phase becomes a real `<form action={refreshTodayAction}>` in Phase 8, where `refreshTodayAction` calls the configured LLM provider, validates the result with the same `DailyContentSchema`, and calls `setDailyContent(userId, iso, generated, 'llm')`. Because validation is centralised in `setDailyContent`, Phase 8 inherits the same human-readable error messages for free — a malformed LLM response surfaces as "god.prayer: Required" in the UI just like a malformed paste.

6. **Phase 10 (scheduler) hand-off.** Phase 10's per-user cron job (`node-cron` in dev, Railway cron in prod) targets `setDailyContent(userId, todayISO(), generated, 'llm')` for every active user just before their configured refresh hour. It reads the `RefreshLog` table (Phase 10 introduces it) to dedupe; `DailyContent` itself is naturally idempotent because of the `(userId, iso)` unique constraint — a duplicate cron tick for the same day overwrites the row, which is the desired refresh-in-place behavior anyway. Phase 10 does not need any further changes to `daily-content.ts`.

7. **Past-day editing is still forbidden in v1.** Phase 11 introduces calendar navigation; the contract is that past days are read-only. The hand-edit modal in this phase is hard-coded to `iso = todayISO()`. When Phase 11 lands, gate the topbar `EditContentLink` behind `iso === todayISO()`. Don't try to generalise it now.

8. **Schema evolution.** If a future phase adds a new required field to `DailyContentSchema` (e.g. Phase 11 adds a `weekend` block), all existing rows become "stale" — they validate-fail on read and degrade to the fixture. That's acceptable for v1 (the user simply re-runs Refresh tomorrow), but the migration plan should call it out explicitly. For optional fields, add `.optional()` and existing rows continue to validate — prefer that path when the new field is non-essential.

9. **No emojis. Inline SVG only.** This plan ships no SVGs because the modal uses typographic characters (`×`, `›`) only — same convention as Phase 3. If you find yourself reaching for a unicode emoji during implementation, reach for an inline `<svg viewBox="0 0 24 24" stroke-width="1.6">` instead.

10. **Rate limiting is Phase 8.** Phase 6 places no cap on hand-edits — a user could click Save a thousand times locally and we don't care. Phase 8's "3 manual refreshes per day" cap applies only to LLM-backed refreshes, not to hand-edits, because hand-edits cost nothing and an LLM call costs tokens.

11. **Audit logging is Phase 10.** Every `setDailyContent` call from Phase 10 gets a `RefreshLog` entry; Phase 6's hand-edit path does not (yet). If you want symmetry, Phase 10 can extend the action with an optional `auditTag` parameter — but do not add the `RefreshLog` table in this phase.
